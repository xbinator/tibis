# BMessage 批量渲染性能设计

## 背景

`BMessage` 负责聊天正文、思考内容、Widget 帮助信息等 Markdown 与纯文本内容的解析和渲染。当前实现对每个组件实例同步执行内容解析，并通过递归的 `BlockNode`、`InlineNode` 组件树渲染 Markdown AST。代码块组件还会为每个实例创建一套 `lowlight` 语法注册表并同步执行高亮。

单条消息的成本通常不明显，但批量挂载历史消息或连续接收流式内容时，这些工作会集中在同一轮主线程任务中。实测 100 次 `createLowlight(common)` 约耗时 74ms，同一批测试内容执行 100 次代码高亮约耗时 113ms，Markdown 词法解析约耗时 43ms；这些数据还不包含 Vue 递归组件实例化和 DOM 更新成本。

此前在 `BubblePartThinking` 中将内容降级为纯文本，只是绕开 `BMessage`，同时损失了 Markdown、代码高亮、公式和链接体验。性能问题属于通用渲染器，应在 `BMessage` 内解决。

## 目标

- `BMessage` 批量挂载时不在单个主线程任务中完成全部解析和渲染工作。
- 连续内容更新只处理每个组件实例的最新内容，不重复解析已过期版本。
- 减少 Markdown AST 对应的 Vue 组件实例数量。
- 所有代码块共享同一个 `lowlight` 实例。
- 完成态内容继续支持现有 Markdown、代码高亮、公式、图片、链接、表格和安全行内 HTML。
- `BubblePartThinking` 恢复默认展开的完整 Markdown 展示，不包含专用性能策略。
- 保持 `BMessage` 现有公共 Props 和调用方式兼容。
- 使用确定性的行为测试验证调度边界，避免依赖容易波动的固定耗时断言。

## 非目标

- 不在本次改动中虚拟化整个聊天消息列表。
- 不把 Markdown 解析迁移到 Web Worker。
- 不改变聊天历史分页、折叠状态或滚动交互。
- 不将思考内容降级为纯文本或 Markdown 子集。
- 不引入新的 Markdown、高亮或虚拟列表依赖。
- 不重写 `marked` 解析器及其扩展语法。

## 方案比较

### 方案一：只复用代码高亮器

将 `lowlight` 调整为模块级单例，并避免对未闭合代码围栏反复高亮。改动小，能够消除明确的重复初始化成本，但无法解决批量同步解析和递归 Vue 组件挂载形成的长任务。

### 方案二：优化 BMessage 核心渲染链路

在 `BMessage` 内增加统一帧预算调度，合并同一实例的连续更新；使用单个轻量节点树渲染器生成原生 VNode，只为代码、公式和图片等重型交互节点保留独立组件；同时复用代码高亮器。这一方案同时降低峰值阻塞时间和总实例化成本，并让所有调用方受益。

### 方案三：聊天列表虚拟化

只挂载视口附近的消息可以显著降低超长会话的 DOM 数量，但需要重新处理动态高度、历史消息前插、折叠展开、图片和公式异步高度、反向滚动与锚点补偿。它不能优化 Widget 帮助等其他 `BMessage` 调用方，也没有消除单个富文本实例内部的重复成本。

本次采用方案二。列表虚拟化可以在超长会话仍有独立性能证据时另行设计。

## 架构

优化拆分为三个相互独立的内部单元：

1. 帧预算调度器负责控制解析任务何时执行。
2. 节点树渲染器负责用较少的 Vue 组件实例生成与当前结构等价的 VNode。
3. 代码高亮 helper 负责复用 `lowlight` 并处理完整与未完整代码块。

这些能力属于 `src/components/BMessage`，不向 BChat 暴露 thinking 专用开关。

### 帧预算调度器

新增 BMessage 内部调度模块，维护高优先级和普通优先级队列。队列任务使用组件实例级 token 作为键；同一 token 再次入队时替换旧任务，因此连续流式更新只解析最新内容。

调度规则：

- 使用 `requestAnimationFrame` 作为批次边界。
- 每帧按高优先级、普通优先级顺序执行任务；高优先级队列优先执行最近晋级的任务，使聊天底部和刚进入预加载区的内容先展示。
- 单帧解析预算默认设为 6ms，且每帧最多执行 4 个任务；即使任务本身很快，也不会在一次绘制前集中提交全部 Vue DOM 更新。
- 达到预算后把剩余任务留到下一帧，不在当前帧继续清空队列。
- 当前已处于或接近可见区域的 BMessage 使用高优先级。
- 流式内容更新使用高优先级，并由 token 替换机制自动合并帧内增量。
- 视口外批量挂载的完成态内容使用普通优先级。
- 组件卸载时取消对应 token 的待执行任务。
- 非浏览器测试环境缺少 `requestAnimationFrame` 时，回退到短定时任务。

`BMessage` 的根节点在解析前仍然存在。组件挂载后查找最近的垂直滚动容器，根据该容器而非整个浏览器视口计算优先级。IntersectionObserver 使用一个滚动视口高度的预加载范围；尚未解析的普通任务进入该范围时会重新入队为高优先级。单个正常可见的 BMessage 会在下一次绘制前以高优先级处理；批量历史内容则跨帧完成。

调度器只负责安排任务，不持有 Vue 状态。任务抛错时调度器隔离该任务并继续处理队列，避免一个异常阻塞其他消息。

### BMessage 数据流

现有 Props 监听保留，但不再直接同步调用 `parseMessageNodes`：

```text
props 变化
  -> 生成当前内容快照
  -> 以实例 token 入队或替换待执行任务
  -> 调度器在帧预算内执行最新任务
  -> parseMessageNodes 生成 AST
  -> shallowRef 替换解析结果
  -> 节点树渲染器更新 DOM
```

每个任务捕获 `content`、`type` 和 `loading` 快照。任务执行前再次确认快照仍为最新版本，避免已经开始排队但随后失效的任务覆盖新内容。解析失败时保留上一次成功结果；首次解析失败则使用纯文本节点展示原始内容，确保消息不会空白。

`loading` 仍控制流式光标和尾节点 ID。外部调用方式不变，现有传入 `loading` 的组件继续获得流式语义。调度器本身会合并任何高频 Props 更新，不要求调用方额外实现节流。

### 轻量节点树渲染器

当前 `BMessage` 为每个块节点创建 `BlockNode` 组件，并为大量行内节点递归创建 `InlineNode` 组件。优化后改为单个内部节点树渲染器：

- 段落、标题、列表、引用、表格、分割线和普通行内格式直接生成原生 VNode。
- 递归发生在纯渲染函数中，不为每个 AST 节点创建 Vue setup、Props、inject 和 computed 状态。
- 继续使用 AST 中的稳定 ID 作为块级 key。
- 代码块继续使用 `CodeBlockNode`，保留复制和 Mermaid 功能。
- 行内与块级公式继续使用 `MathNode`、`MathBlockNode`。
- 图片继续使用 `ImageNode`，保留预览索引和点击行为。
- 链接继续调用现有导航 handler，并保持安全 HTML 白名单行为。
- 流式光标、CSS 类名和 Markdown 样式保持现有结构兼容。

`BMessage` 继续提供现有图片预览与导航上下文。旧的 `BlockNode.vue` 和 `InlineNode.vue` 在新渲染器覆盖全部行为并通过测试后移除，避免并存两套递归渲染实现。

### 共享代码高亮器

将 `createLowlight(common)` 从 `CodeBlockNode` 实例 setup 中移到独立模块，整个 renderer 进程只创建一次高亮器。

高亮规则：

- 已闭合代码围栏继续按现有语言别名执行完整语法高亮。
- 未闭合的流式代码围栏直接渲染安全纯文本节点，不在每个增量上重新运行 lowlight。
- 围栏闭合后执行一次高亮并切换到最终高亮结构。
- 完成态高亮结果按规范化语言和代码内容写入最多 100 项的 LRU 缓存；后续流式快照重建相同代码块时直接复用安全节点。
- 未注册语言和高亮异常继续回退到纯文本代码节点。
- Mermaid 完成态仍走现有 Mermaid 预览逻辑，不改变其动态导入和错误展示。

最终完成态的视觉与交互保持不变。唯一允许的流式差异是未闭合代码块在生成期间暂不显示语法颜色，闭合后立即恢复完整高亮。

## 兼容性

`BMessageProps` 不新增必填字段，也不删除现有字段。以下调用方无需引入性能配置：

- BChat 助手正文
- BChat 思考内容
- BChat 工具纯文本代码
- 编辑器 AI 选择预览
- Widget Action 与 Schema 帮助

`BubblePartThinking` 只需恢复 `type="markdown"`。默认展开状态仍由现有 `BubblePart` 配置控制，不进入 `BMessage` 优化范围。

## 测试策略

### 调度器单元测试

- 同一 token 多次入队只执行最新任务。
- 高优先级任务先于普通任务执行。
- 达到帧预算后剩余任务进入下一帧。
- 单个任务超过预算时队列仍能继续推进。
- 取消 token 后任务不执行。
- 某个任务抛错不阻塞后续任务。
- 无 `requestAnimationFrame` 时回退调度仍可完成。

### BMessage 组件测试

- 单个可见实例继续展示完整 Markdown。
- 批量挂载多个实例后，第一帧不会执行全部解析任务；刷新后最终全部展示。
- 连续更新 content 时只提交最新解析结果。
- 组件卸载不会提交待执行结果。
- 解析失败保留旧内容或展示首次纯文本回退。
- 现有图片、链接、列表、表格、公式、代码复制和 Mermaid 测试继续通过。

### 节点渲染等价测试

- 对已有 parser 测试覆盖的每类 AST 节点断言最终标签、文本、属性和事件行为。
- 验证普通块和行内节点不再创建递归 `BlockNode` / `InlineNode` 组件实例。
- 验证稳定 key 和流式光标结构未回退。

### 高亮测试

- 多个代码块只初始化一次 lowlight。
- 未闭合代码围栏不调用 `lowlight.highlight`。
- 围栏闭合后执行高亮。
- 未注册语言和高亮异常回退纯文本。

性能回归测试使用“每帧执行任务数量、过期任务是否被替换、组件实例结构”等确定性指标，不把 CI 机器上的绝对毫秒数作为通过条件。开发阶段保留批量挂载基准数据，用于比较优化前后趋势。

## 验收标准

- `BubblePartThinking` 默认展开并展示完整 Markdown。
- 批量挂载 BMessage 不会在同一帧同步解析全部实例。
- 连续更新不会解析被后续内容替换的中间版本。
- 完成态 Markdown 功能与现有测试表现一致。
- 代码高亮器在模块生命周期内只初始化一次。
- 普通 Markdown 块和行内节点不再各自创建 Vue 组件实例。
- BMessage、BChat 相关测试、TypeScript、ESLint 和 Stylelint 检查通过。
- 不新增 BChat 或 thinking 专用性能 Props。
