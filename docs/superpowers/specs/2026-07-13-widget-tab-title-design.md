# Widget 保存后同步标签标题设计

## 背景

Widget 页面当前使用文件标题创建顶部标签。页面设置中的“名称”写入 `WidgetData.name`，保存成功后没有同步到标签。WebView 已通过标题同步 hook 调用 `tabsStore.updateTabTitle`，Widget 页面需要提供一致体验，同时保证标签只展示真正保存成功的名称。

## 目标

- Widget 名称实际保存成功后，标签显示去除首尾空白的名称。
- 名称为空、仅包含空白或保存内容无效时，回退显示文件标题。
- 尚未保存、保存取消或保存失败时，标签保持上一次实际保存的标题。
- 手动保存、失焦保存、内容变更自动保存和外部文件更新使用同一保存基线。
- 写盘期间继续编辑时，标签显示本次真正写入磁盘的名称，后续内容继续保持 dirty。
- 并发触发保存对话框时只执行一个完整保存事务，不允许后续调用发布未写盘快照。

## 非目标

- 不在用户输入名称时立即更新标签。
- 不修改文件名、文件路径或最近文件记录的名称字段。
- 不改变 HeaderTabs 的渲染、图标或关闭行为。
- 不改变保存策略的触发时机。

## 方案

`useFileSession` 已维护内部 `savedContent`。将它作为只读语义的 Ref 暴露给页面，并确保保存时使用写盘前捕获的内容快照更新该基线：

1. 磁盘保存和另存为在调用原生 API 前捕获 `contentToSave`。
2. 写盘成功后将 `contentToSave` 设为 `savedContent`，而不是读取可能已经变化的当前内容。
3. 如果当前内容仍等于快照则清除 dirty；如果写盘期间又有修改则继续标记 dirty。
4. 保存对话框以完整事务 Promise 做 single-flight，后续并发调用复用同一事务结果。
5. Widget 页面从只读的 `session.savedContent` 解析已保存的 `name`，空名称或无效 JSON 回退 `session.currentTitle`。
6. 初始 `addTab` 只负责标签身份、路径和缓存，并保留已有标题；`useWidgetTabTitle` 负责后续 `tabId`、保存标题或原始保存基线变化。即使派生标题文本相同，加载出的新保存基线也会执行一次权威同步。

该设计以“实际保存内容”而非 dirty 状态推断作为唯一标题来源，因此不会被异步加载、文件重命名或并发编辑绕过。

## 数据流

1. 用户修改 Widget 名称，当前内容改变并进入 dirty，`savedContent` 不变，标签不变。
2. 保存入口捕获当前内容快照并写盘。
3. 写盘成功后保存基线更新为该快照。
4. Widget 标题 computed 从新基线解析名称。
5. 标题同步 hook 更新当前标签。
6. 如果写盘期间又有修改，当前内容与保存快照不同，dirty 保持为 true，后续名称不会提前显示。

## 异常与边界

- 保存对话框取消、序列化失败或磁盘写入失败：不更新 `savedContent`，标签不变。
- 名称为空或仅包含空白：显示文件标题。
- 已保存 JSON 无法解析或 `name` 不是字符串：显示文件标题。
- 外部文件变化：外部内容同时成为当前内容和保存基线，标签同步外部文件中的名称。
- 文件重命名：保存基线有名称时继续显示该名称；无名称时 computed 随文件标题更新。
- 并发保存对话框：后续调用复用首个完整事务，不重复打开对话框或发布另一份快照。

## 测试

- 保存基线名称变化后更新标签。
- 保存名称为空或 JSON 无效时回退文件标题。
- 页面接收到新的 `savedContent` 后更新正确的标签 ID。
- 写盘期间再次编辑时，保存基线保持为实际写盘快照，当前新内容继续 dirty。
- 并发另存时原生保存只调用一次，后续内容不被误标记为已保存。
- 初始标签创建不重复调用标题更新，`tabId` 改变时同步当前保存标题。
- 保存基线变化但派生标题文本相同时，仍覆盖历史遗留的旧标签标题。
- Widget 页面现有交互回归测试继续通过。
