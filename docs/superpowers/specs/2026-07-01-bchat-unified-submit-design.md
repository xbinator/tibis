# BChat Unified Submit Design

## Goal

Refactor message-level interactive components so they adapt their own output into a unified submit action. The top-level `src/components/BChat/index.vue` should receive one normal submit event and should not branch on whether the submission came from `QuestionCard.vue` or `BubblePartWidget.vue`.

## Current State

`src/components/BChat/components/QuestionCard.vue` emits a `runtime-input` payload with `kind: 'user_choice'`.

`src/components/BChat/components/MessageBubble/BubblePartWidget.vue` emits a `runtime-input` payload with `kind: 'widget_result'`.

`src/components/BChat/index.vue` currently owns separate handlers for these shapes and dispatches with an explicit `kind` branch:

- `handleChatUserChoiceSubmit`
- `handleChatWidgetSubmit`
- `handleChatRuntimeInput`

This makes the top-level chat component aware of message-part-specific submission details.

## Target Design

Introduce a unified BChat submit action type for message-level interactive submissions. The action should be prepared by the leaf component that owns the UI interaction and should run itself through a submit context.

`QuestionCard.vue` will convert selected answers and optional supplementary text into a user-choice submit action.

`BubblePartWidget.vue` will convert widget runtime output into a widget-result submit action, including the structured `widget_result` message part needed by runtime send.

`MessageBubble.vue` and `ConversationView.vue` will only forward a `submit` event. They should not interpret payload contents.

`src/components/BChat/index.vue` will keep a single submit entry for message-level interactive submissions. It can provide a submit context, but neither the component nor the submitter should branch on a payload discriminator.

## Unified Payload Type

Add a BChat-local submit action type in `src/components/BChat/utils/submitAction.ts`. This type belongs to the BChat component boundary, not to shared `types/chat.d.ts`, because it describes renderer component communication rather than a persisted chat record or Electron runtime API.

```typescript
/**
 * BChat message-level interactive submit action.
 */
export interface SubmitAction {
  /** Run the already-adapted submit action. */
  run: (context: SubmitContext) => Promise<void>;
}

/**
 * Runtime capabilities exposed to executable submit actions.
 */
export interface SubmitContext {
  /** Continue an awaiting assistant turn with an answer generated by QuestionCard. */
  continueAssistantTurn: (answer: AIUserChoiceAnswerData) => Promise<void>;
  /** Send a structured user message generated by a leaf component. */
  sendAdaptedUserMessage: (input: AdaptedUserMessageInput) => Promise<void>;
}

/**
 * Input for sending a renderer-created structured user message.
 */
export interface AdaptedUserMessageInput {
  /** Renderer-created user message. */
  userMessage: Message;
  /** Runtime input parts matching the user message. */
  parts: ChatRuntimeUserInputPart[];
  /** Error message used if runtime send fails before a persisted error can be written. */
  errorMessage: string;
  /** Whether to clear the main input draft after sending. */
  clearDraft?: boolean;
}
```

`QuestionCard.vue` emits `createUserChoice(answer)`.

`BubblePartWidget.vue` converts widget output into `ChatMessageWidgetResultPart`, creates the matching renderer user message, and emits `createRuntimeUserMessageSubmitAction(...)`.

The top-level component accepts `SubmitAction` and calls the unified submitter. The submitter calls `action.run(context)` and does not inspect a discriminator field.

## Submitter Hook

Add `src/components/BChat/hooks/useChatSubmitter.ts` to own the runtime-specific actions. `src/components/BChat/index.vue` wires dependencies once and calls only `submit(action)`.

```typescript
/**
 * Unified submitter dependencies supplied by BChat index.
 */
export interface UseChatSubmitterOptions {
  /** Unified chat task runtime. */
  taskRuntime: Pick<ReturnType<typeof useChatTaskRuntime>, 'activeTask' | 'beginTask' | 'finishTask'>;
  /** Current active session ID. */
  getSessionId: () => string | undefined;
  /** Runtime request config resolver. */
  resolveRuntimeRequestConfig: () => Promise<ChatRuntimeRequestConfig | null>;
  /** User-choice runtime continuation. */
  submitUserChoice: (input: BChatRuntimeSubmitUserChoiceInput) => Promise<ChatRuntimeStartResult>;
  /** Sends a renderer-created user message through the normal runtime send path. */
  sendRuntimeUserMessage: (input: AdaptedUserMessageInput) => Promise<void>;
}

/**
 * Unified message-level interactive submitter.
 */
export interface UseChatSubmitterReturn {
  /** Submit an already-adapted message-level interaction action. */
  submit: (action: SubmitAction) => Promise<void>;
}
```

The hook provides a submit context with:

- `continueAssistantTurn(answer)` preserving the existing `chatRuntime.submitUserChoice` semantics.
- `sendAdaptedUserMessage(input)` delegating to the existing runtime user-message send path.

`submit(action)` should only call `action.run(context)`.

## Runtime Semantics

User-choice submissions must still use the existing runtime user-choice continuation semantics. They are not equivalent to ordinary text messages because the main process updates the pending assistant tool result before resuming the stream.

Widget-result submissions can continue to flow through runtime send as a structured user message whose part is `type: 'widget_result'`.

The unified submit layer should hide this difference from the top-level component API, not erase the underlying runtime behavior.

## Component API

Replace the message interaction event chain:

- `runtime-input`

With:

- `submit`

The event value should be a local BChat submit action, not the current `ChatMessageRuntimeInput` union exposed from `types/chat.d.ts`.

The concrete chain after migration is:

- `QuestionCard.vue` emits `submit` with an executable `SubmitAction`.
- `BubblePartWidget.vue` emits `submit` with an executable `SubmitAction`.
- `MessageBubble.vue` forwards `submit` from both child components and keeps `widget-part-change` unchanged.
- `ConversationView.vue` forwards `submit` from `MessageBubble.vue`.
- `src/components/BChat/index.vue` receives `@submit="chatSubmitter.submit"`.

No intermediate component should parse or transform the action, and the unified submitter should not branch on the action shape.

## Migration Strategy

Use a hard switch inside `src/components/BChat`. Do not support both `runtime-input` and `submit` in parallel, because the old event name encourages the old top-level dispatch shape.

After the migration:

- `rg "runtime-input" src/components/BChat test/components/BChat` should return no matches.
- Existing tests should be updated in the same change.
- The public runtime APIs in `types/chat-runtime.d.ts` stay unchanged.

## Old Type Policy

Remove these old component-event types from `types/chat.d.ts` if no non-BChat consumers remain:

- `ChatMessageUserChoiceRuntimeInput`
- `ChatMessageWidgetResultRuntimeInput`
- `ChatMessageRuntimeInput`

Keep the lower-level data types that are still part of chat records or runtime APIs:

- `AIUserChoiceAnswerData`
- `ChatMessageWidgetSubmitPayload`
- `ChatMessageWidgetResultPart`
- `ChatMessageWidgetSubmitResult`

If a non-BChat consumer appears during implementation, keep a deprecated compatibility alias in `types/chat.d.ts` and document the remaining consumer in the implementation notes.

## Error Handling

The unified submit path must preserve current behavior:

- Busy chat tasks should reject new submissions without starting another task.
- Missing session or model configuration should finish the task and avoid leaving loading state stuck.
- Widget submission failures should still show/persist a runtime error message where appropriate.
- User-choice completion should finish the task when runtime reports `completed: true`.

## Testing

Update existing BChat tests that assert `runtime-input` emissions so they assert `submit` emissions with the new adapted action shape.

Keep coverage for:

- `QuestionCard.vue` answer and cancel submissions execute `continueAssistantTurn(answer)` through the submit context.
- `BubblePartWidget.vue` widget output conversion executes `sendAdaptedUserMessage(input)` through the submit context.
- `MessageBubble.vue` and `ConversationView.vue` event forwarding.
- `src/components/BChat/index.vue` using one submit entry without the removed handlers.

## Implementation Order

1. Add `src/components/BChat/utils/submitAction.ts` and `src/components/BChat/hooks/useChatSubmitter.ts`.
2. Update `QuestionCard.vue` and its tests to emit executable `submit` payloads.
3. Update `BubblePartWidget.vue` and its tests to emit executable `submit` payloads.
4. Update `MessageBubble.vue` and `ConversationView.vue` to forward only `submit` for message-level interactions.
5. Update `src/components/BChat/index.vue` to wire `useChatSubmitter` and remove `handleChatUserChoiceSubmit`, `handleChatWidgetSubmit`, and `handleChatRuntimeInput`.
6. Remove or deprecate the old `ChatMessageRuntimeInput` family according to the old type policy.
7. Update BChat tests and changelog.
8. Run focused BChat tests, then TypeScript checking.

## Out Of Scope

This refactor does not change widget mounted lifecycle handling or `widget-part-change` persistence.

This refactor does not change the main-process runtime APIs.
