# 助手消息创建会话分支设计

## 背景

`src/components/BChat/components/MessageBubble.vue` 的助手消息工具栏当前提供复制与重新生成操作。用户需要在同一工具栏中增加“创建分支”按钮，从选中的已完成助手消息处复制一份独立会话，并立即切换到新会话继续聊天。

当前聊天消息的数据库主键在所有会话间全局唯一，长会话还可能包含指向消息 ID 的压缩边界与独立压缩记录。创建分支不能直接复用原消息主键，也不能让新分支的回退或压缩操作修改源会话记录。因此，分支创建应由主进程在单一数据库事务内完成。

## 目标

- 在已完成助手消息的现有工具栏中增加“创建分支”按钮。
- 复制源会话开头到目标助手消息为止的完整历史，包含目标消息。
- 新分支沿用源会话当前标题，不添加编号或其他后缀。
- 创建成功后刷新会话历史并立即切换到新分支。
- 重建全局唯一标识和跨消息引用，使新旧会话相互独立。
- 任一步骤失败时整体回滚，不留下空会话或残缺消息。

## 非目标

- 不展示源会话与分支会话之间的树形关系。
- 不给 `ChatSession` 增加父会话或根会话字段。
- 不复制源会话的 Todo 运行态快照。
- 不增加新的“生成中是否允许创建分支”规则；按钮继续由现有助手消息工具栏的显示条件控制。
- 不自动重新生成目标助手回答，也不自动提交新的用户消息。

## 交互设计

助手消息工具栏在复制按钮之后、重新生成按钮之前增加使用 `lucide:git-branch-plus` 图标的文本按钮，提示文字为“创建分支”。按钮仅随现有 `showAssistantToolbar` 条件出现，即消息角色为 `assistant` 且 `finished === true`。

创建分支按钮不额外显示加载态或禁用态。重复点击事件由 `BChat` 的底层请求标记统一拦截；创建成功后切换到新会话，创建失败时通过现有交互 Toast 显示错误，当前会话保持不变。

## 组件与数据流

1. `MessageBubble.vue` 发出携带目标 `Message` 的 `branch` 事件，不持有请求状态。
2. `ConversationView.vue` 将目标消息原样转发给 `BChat`。
3. `BChat/index.vue` 使用内部消息 ID 拦截重复请求，并通过 `asyncTo` 调用会话 Store 的分支方法。
4. 会话 Store 通过 Electron API 发送源会话 ID 和目标消息 ID。
5. 主进程聊天服务在事务内验证源数据、创建新会话、复制消息及相关压缩记录。
6. 主进程返回新 `ChatSession` 后，`BChat` 沿用现有 `session-created` 事件通知 `ChatSider`。
7. `ChatSider` 沿用现有会话创建处理流程，设置当前会话、刷新 `SessionHistory` 并显示新分支。

该数据流不要求 `MessageBubble` 了解会话持久化，也不把消息复制逻辑放入 `ChatSider`。

## 主进程接口

新增一个语义完整的会话分支接口：

```typescript
chatSessionBranch(sourceSessionId: string, targetMessageId: string): Promise<ChatHandlerResult<ChatSession>>
```

主进程服务提供对应的同步事务方法。IPC handler 只负责参数转发和统一错误包装，不在 handler 内拆分多次写操作。

## 事务与复制规则

### 源数据校验

事务开始后必须验证：

- 源会话存在且类型为 `assistant`。
- 目标消息属于源会话。
- 目标消息角色为 `assistant` 且已经完成。
- 按聊天消息的既有稳定排序能够找到目标位置。

任一条件不满足时抛出明确错误，并且不写入任何新数据。

### 新会话

- 使用新的会话 ID。
- `type` 与 `title` 原样继承源会话。
- `createdAt`、`updatedAt` 和 `lastMessageAt` 使用创建分支时的当前时间，使新分支出现在会话历史顶部。
- `usage` 根据实际复制的消息重新汇总，不能直接继承源会话中目标消息之后的用量。

允许多个会话具有相同标题；标题不承担分支身份或唯一性职责。

### 消息范围与顺序

主进程读取源会话的完整消息集合，使用与历史加载一致的 `createdAt`、角色顺序和消息 ID 稳定排序，截取从第一条消息到目标助手消息的闭区间。复制后的消息保留原创建时间和相对顺序。

### 标识与引用重建

- 每条复制消息生成新的消息 ID，并建立旧消息 ID 到新消息 ID 的映射。
- 每个结构化消息 part 生成新的 part ID。
- 新 ID 工厂拒绝与源会话标识或本次已生成标识重复的结果，防止测试替身或极小概率随机碰撞产生内部覆盖。
- 已知的消息引用字段使用消息 ID 映射更新，包括压缩元数据、compaction part 和 runtime compaction meta 中的消息 ID。
- 复制消息直接引用的压缩记录，以及这些记录通过 `derivedFromRecordId` 递归引用的祖先记录。每条记录生成新的记录 ID；同一源 `recordSetId` 统一映射到同一个新记录集合 ID。
- 压缩记录的 `sessionId` 改为新会话 ID，`derivedFromRecordId`、消息边界、源消息列表和保留消息列表均使用对应的新 ID。只要任一必需记录或消息引用无法映射，就终止并回滚整个事务。
- 消息中的压缩记录引用同步指向新记录，保证新分支回退时不会使源会话压缩记录失效。
- 旧 `runtimeId` 与 `parentRuntimeId` 不复制，避免新会话复用已结束 Runtime 的路由身份。
- 附件 ID、文件路径、工具调用 ID、工具结果和 Widget 自身的 session ID 保持原值。这些字段属于内容或工具内部关系，不能通过字符串替换误改。
- 所有引用替换都针对已知类型字段执行，禁止对序列化 JSON 做无差别文本替换。

### 原子性

新会话、消息和压缩记录必须在同一个数据库事务中使用普通 `INSERT` 写入，禁止使用会静默覆盖旧数据的 `INSERT OR REPLACE`。发生 ID 冲突、解析失败或写入异常时回滚整个事务。创建分支读取消息和复制范围实际引用的压缩记录祖先闭包时使用严格 JSON 映射，只接受当前最新 schema；非空字段损坏或类型无效时直接失败，未被复制范围引用的记录不阻断分支。普通历史展示仍保留既有容错行为。源会话在任何路径下都不得被更新。

## 状态与错误处理

`BChat` 使用目标消息 ID 表示正在进行的分支请求。一次请求完成前忽略后续分支事件，结算后清空标记；按钮本身不绑定加载或禁用状态。Store Promise 通过 `asyncTo` 转换为错误优先元组，失败时不发送 `session-created`。

Store 使用现有数据库初始化重试和 `unwrap` 机制。错误最终由 `BChat` 转换为用户可读 Toast；失败时不发送 `session-created`，因此 `ChatSider` 不会切换会话。

## 测试设计

### 组件测试

- 未完成或非助手消息不显示创建分支按钮。
- 已完成助手消息显示带正确图标和提示的按钮。
- 点击后 `MessageBubble`、`ConversationView` 和 `BChat` 正确传递目标消息。
- 按钮不绑定分支请求加载或禁用状态，重复事件仍不会发起第二次 Store 请求。
- 成功时发送新会话事件，失败时保留当前会话并显示错误。

### Store 与接口测试

- Store 正确传递源会话 ID 和目标消息 ID，并解包返回的新会话。
- preload 和 Electron API 类型声明保持一致。
- IPC handler 将参数传递给聊天服务并统一包装成功或失败结果。

### 主进程服务测试

- 新会话标题与源标题完全一致。
- 复制范围包含目标助手消息，但不包含其后的消息。
- 新消息 ID 和 part ID 与源数据不冲突，消息内部引用完成重建。
- 附件、工具调用与 Widget 内容保持不变。
- 复制的压缩记录及其消息引用只指向新会话数据。
- 用量仅汇总复制范围，新会话按当前时间进入历史顶部。
- 从已经复制产生的会话再次创建分支时，标题仍保持不变。
- 源会话不存在、目标不属于源会话、目标不是已完成助手消息时事务不写入任何数据。
- 模拟中途写入失败时，新会话、消息和压缩记录全部回滚，源会话不变。
- 使用 Electron ABI 对应的真实临时 SQLite 验证全局消息 ID 冲突会回滚新会话，并保持源会话与无关会话数据不变。

## 验收标准

用户点击任一已完成助手消息上的“创建分支”按钮后，会看到一个标题不变的新会话；该会话只包含截至所选助手消息的历史，并可正常继续发送消息、压缩上下文和执行回退。原会话的消息、用量和压缩记录保持不变。任何创建失败都不会产生可见的残缺会话。
