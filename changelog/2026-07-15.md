# 2026-07-15

## Changed

- 移除小组件设置页的重复 Store 初始化，统一由默认布局负责资源扫描与目录监听。
- 将 Skill 与 Widget 资源初始化 Hook 迁移至默认布局目录，明确其应用布局级职责。
- 抽取公共 `src/hooks/useFileStoreInit.ts`，统一 Skill / Widget Store 初始化的「准备屏障 → 订阅事件 → 监听目录 → 扫描 → 清理」流程；`useSkillInit` 与 `useWidgetInit` 改为纯配置委托。
- 将 `src/shared/workspace/pathUtils.ts` 迁移到 `src/utils/file/`：`posix.ts` 补齐 `join` / `basename` / `parseParts` / `slashify`，新增 `path.ts`（路径片段安全校验）与 `workspace.ts`（工作区根解析与包含判断）。所有集合统一为 `export const xxx = { ... }` 形式（仿 `posix`），函数命名进一步收敛：`resolvePathAgainstWorkspace` → `workspace.resolveWithin`，`isPathInsideWorkspace` → `workspace.contains`（参数顺序由 `(target, root)` 调整为 `(root, target)`），`isSafeFilePathSegment` → `path.isValidSegment`，`normalizeSafeRelativeFilePath` → `path.validatePath`，`normalizeSeparators` → `posix.slashify`。
- 资源监听 hook 命名收敛：通用 `useFileStoreInit` / `FileStoreInitConfig` / `createTargetFileMatcher` 重命名为 `useWatchResource` / `WatchResourceConfig` / `createResourceMatcher`，同步重排为 `src/hooks/useWatchResource.ts`；布局级 `useSkillInit` / `useWidgetInit` 同步重命名为 `useWatchSkillResource` / `useWatchWidgetResource`，新文件分别位于 `src/layouts/default/hooks/useWatchSkillResource.ts` 与 `useWatchWidgetResource.ts`。语义强调「监听用户目录中的资源并同步到 Store」，弱化「Store 内部初始化」这一被误读的语义。
- 布局级资源监听 hook 进一步精简命名：`useWatchSkillResource` / `useWatchWidgetResource` → `useWatchSkill` / `useWatchWidget`，文件同步迁移至 `src/layouts/default/hooks/useWatchSkill.ts` 与 `useWatchWidget.ts`；调用方（`src/layouts/default/index.vue`、测试 `ai-resource-init.test.ts`）同步更新。
- `useWatchResource` 内部职责收拢：移除 `createResourceMatcher` 导出，隐藏目录（`.tmp-xxx` / `.bak-xxx` 等）过滤下沉到 hook 内部的 `isHiddenPath`，基于 `config.rootDir` / `config.subDir` 自动判定。调用方 `isTargetFile` 退化为纯业务谓词，无需再传入 `'.agents'` / `'.tibis'` 等内部路径段。`src/layouts/default/hooks/useWatchSkill.ts` 与 `useWatchWidget.ts` 的 `isManagedSkillFile` / `isManagedWidgetFile` 相应精简。
- 资源路径谓词改用正则：`useWatchSkill` 谓词为 `/SKILL\.md$/`；`useWatchWidget` 谓词为 `/\/\.tibis\/widgets\/[^/]+\/widget\.json$/`。隐藏目录过滤仍由 `useWatchResource` 内部 `isHiddenPath` 承担。
- `useWatchResource` 配置项统一改用 `onXxx` 命名：明确其为「由 hook 在特定时机回调」的钩子点。`prepareInitialization` → `onBeforeInitialize`（初始化前建立屏障）、`init` → `onInitialize`（执行扫描）、`finishInitialization` → `onAfterInitialize`（释放屏障）、`handleChange` → `onChange`（处理 add / change / unlink）、`parseFile` → `onParseFile`、`createUnlinkPayload` → `onCreateUnlinkPayload`、`isTargetFile` → `onIsTargetFile`。`src/layouts/default/hooks/useWatchSkill.ts` 与 `useWatchWidget.ts` 调用方同步更新；测试用例不受影响（mock 的是 store 方法，hook 配置项命名变更对测试透明）。
- Store 方法同步收拢为「动词 + 资源」对齐命名：`useSkillStore` / `useWidgetStore` 的 `prepareInitialization` → `beforeInitialize`、`finishInitialization` → `afterInitialize`、`init` → `initialize`。store 内部互调（`initialize` 内部调用 `beforeInitialize` / `afterInitialize`）与 return object 同步更新；调用方 `useWatchSkill` / `useWatchWidget` 与 5 个测试文件（`ai-resource-init.test.ts` / `stores/ai/skill.test.ts` / `stores/ai/widget.test.ts` / `views/settings/tools/widget/index.test.ts` / `components/BChat/session-id-runtime.test.ts`）的 mock 与断言同步更新。
- Store 内部 `applySkillChange` / `applyWidgetChange` 重命名为 `updateSkillChange` / `updateWidgetChange`，store 内全部调用点与 `return` 同步更新。
- Store 内部 `updateSkillChange` / `updateWidgetChange` 的 `findIndex` 三元表达式统一为单 `findIndex` + 早返形式：先按 `filePath` 匹配，unlink 事件则直接结束，非 unlink 事件再按业务键（skill 的 `name` / widget 的 `id`）兜底；语义与原三元完全一致。
- Store `initialize` 内部 `try { await scan… } catch { console.error(...) } finally { afterInitialize() }` 改写为 `await asyncTo(scan…); afterInitialize();`：错误日志由 `src/utils/asyncTo.ts` 内部统一处理（`logger.error`），符合项目异步错误处理规范；`initTaskPromise` 仍返回原 Promise，初始化失败时屏障照常释放。
- `resolveLatestEnabledWidget` / `resolveLatestEnabledSkill` 内层 `try { await readFile … } catch { unlink(...) }` 同样改用 `asyncTo`：失败/空结果时回退为 `unlink` 通知 store，成功时正常执行 `change` / `add+unlink` 分支；错误日志由 `asyncTo` 统一处理。
- 删除 `src/ai/skill/index.ts`：消费方改为直接引用具体文件（`@/ai/skill/parser`、`@/ai/skill/scanner`、`@/ai/skill/types`）；`scanSkills` / `parseSkillMarkdown` / `DEFAULT_SKILL_MAX_CONTENT_LENGTH` 等导出按来源拆分为多个 import 语句。
- 修复 `src/ai/skill/parser.ts` 错误转出口 `export { join as joinPath } from '@/utils/file/posix'`：`@/utils/file/posix` 仅导出 `posix` 聚合对象，不存在 `join` 具名导出，TS 编译会失败。移除该行后，`scanner.ts` 与 `SkillCreator.vue` 改为直接 `import { posix } from '@/utils/file/posix'` 并使用 `posix.join(...)`，与文件内 `posix.dirname` 用法保持一致。
- 删除 `src/ai/widget/index.ts`：消费方改为直接引用具体文件（`@/ai/widget/parser`、`@/ai/widget/scanner`、`@/ai/widget/importer`、`@/ai/widget/types`）；`importWidgetJsonFile` / `importWidgetZipFile` / `parseWidgetJson` / `scanWidgets` / `WidgetDefinition` 等导出按来源拆分为多个 import 语句。
- 修复 `src/ai/widget/parser.ts` 同样的错误转出口 `export { join as joinPath } from '@/utils/file/posix'`：移除该行后，`scanner.ts` 与 `WidgetCreator.vue` 改为直接 `import { posix } from '@/utils/file/posix'` 并使用 `posix.join(...)`，与文件内 `posix` 现有用法保持一致。
- 修复 `src/ai/tools/shared/fileTool.ts` 残留旧 API：`isPathInsideWorkspace` → `workspace.contains` 重命名时漏改一处调用点，85 行直接调用未导入的 `contains` 导致 `tsc --noEmit` 报错 `TS2304: Cannot find name 'contains'`。补回 `workspace.` 前缀，与同文件 77/92 行的 `workspace.isAbsoluteFilePath` / `workspace.resolveWithin` 用法保持一致。
